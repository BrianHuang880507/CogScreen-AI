# Agent.md — CogScreen-AI 專案更新執行指引

## 專案背景
本專案為高齡者認知互動遊戲平台，前端以 **HTML / CSS / JavaScript** 為主，強調：
- 操作簡單
- 畫面清楚
- 適合長者使用
- 遊戲與問答可作為認知刺激或初步觀察工具

本次更新重點是調整 **SPMSQ 問題內容、遊戲 UI、節奏、評分機制與獎勵機制**，讓整體體驗更符合高齡者使用情境。

---

## 本次必做需求

### 1. SPMSQ 題目調整
將原本偏向「時事」的題目，改為「一般常識問題」。

#### 需求說明
- 避免使用會因時間而過期的新聞、近期事件、流行話題。
- 題目應偏向穩定、不易過時的常識內容。
- 題目難度要適合長者，不宜過度刁鑽。
- 儘量維持簡單、直白、口語化。

#### 題型建議
可從以下方向改寫：
- 日期與時間基本概念
- 台灣生活常識
- 基礎數字概念
- 常見動物、植物、食物
- 一般社會生活知識
- 簡單方向、季節、節日概念

#### 實作要求
- 若 SPMSQ 題庫為獨立 JSON / JS 陣列，直接更新題目來源。
- 若題目邏輯寫死在 JS，請整理成可維護的題庫結構。
- 保留未來可再擴充題目之彈性。

#### 驗收標準
- 不再出現任何明顯時事型題目。
- 題目內容可離線使用，不依賴外部 API 或即時資料。
- 題目語句簡短清楚。

---

### 2. 遊戲畫面放大與版面整合
遊戲畫面需要放大，最好接近全畫面呈現；標題與返回選單按鈕直接整合在遊戲畫面上，不要額外切出一個很明顯的上方區塊。

#### 需求說明
- 遊戲主區域應優先占滿視窗。
- 減少不必要留白與上下切割感。
- 標題與返回按鈕可採疊加式或浮動式放在遊戲畫面上方。
- 需兼顧長者可讀性與可點擊性。

#### UI 原則
- 字體與按鈕需放大。
- 遊戲互動元素要更醒目。
- 優先考慮桌機全畫面顯示，也要避免在平板上破版。
- 不要讓 header 區塊佔掉太多可玩的空間。

#### 實作要求
- 檢查所有遊戲頁面共用 layout。
- 若有共用 header / game container / top bar，請統一調整。
- 優先用 CSS 版面調整完成，不要硬編碼絕對尺寸到無法維護。
- 返回按鈕需保持清楚、固定、容易點擊。

#### 驗收標準
- 遊戲畫面視覺上明顯比目前大。
- 標題與返回按鈕整合於遊戲畫面內，不再像獨立 block。
- 主要互動區在常見螢幕尺寸下可優先完整呈現。

---

### 3. 打地鼠節奏放慢，且出洞前要有提示
目前打地鼠對老人家而言仍偏快，需要顯著放慢；地鼠出洞前要先有提示，幫助使用者預備反應。

#### 需求說明
- 地鼠出現頻率降低。
- 地鼠停留時間可延長。
- 地鼠出洞前增加提示效果。

#### 提示效果建議
可擇一或混合使用：
- 洞口先閃爍
- 洞口邊框亮起
- 洞口先浮現陰影／預告標記
- 地鼠出現前 0.5～1.5 秒先有視覺提示

#### 實作要求
- 將速度參數抽出成可調整設定值。
- 至少區分：
  - 提示時間
  - 地鼠顯示時間
  - 兩次出現間隔
- 避免提示與真正可點擊狀態混淆。
- 若有音效，提示音需柔和且不刺耳；若目前沒有音效，可先只做視覺提示。

#### 驗收標準
- 體感速度明顯比原本慢。
- 地鼠正式出現前，使用者可看到清楚提示。
- 提示與出現狀態在視覺上可區分。

---

### 4. 以時間作為評分機制的遊戲，延長至分鐘等級
所有以「時間」作為核心評分或限制機制的遊戲，應從目前較短秒數調整為分鐘等級，避免長者操作壓力過高。

#### 需求說明
- 將短秒數倒數改為較寬鬆的分鐘級設定。
- 視遊戲性質可設定為：
  - 1 分鐘
  - 2 分鐘
  - 3 分鐘
  - 或可配置式參數

#### 實作要求
- 把時間設定集中管理，不要散落各檔案。
- 畫面顯示格式建議改為 `mm:ss`。
- 若既有邏輯直接以秒為單位，也可內部保留秒數運算，但 UI 顯示需改為分鐘格式。
- 若某些遊戲不適合硬性倒數，可評估改為「完成耗時紀錄」而不是強制快速作答。

#### 驗收標準
- 時間型遊戲不再只給短秒數壓迫作答。
- 使用者可以清楚看到分鐘級時間資訊。
- 時間設定具可維護性。

---

### 5. 加入獨立得分機制與可兌換獎勵概念
目前評分機制與遊戲通關或既有結果邏輯不同，現在需要額外加入「得分／點數」系統，可作為鼓勵使用者持續遊玩的回饋機制，例如兌換虛擬獎勵。

#### 需求說明
- 新增一套與既有評分不同的「遊戲點數」或「獎勵分數」機制。
- 遊戲完成、答對、連擊、完成關卡等行為可獲得點數。
- 點數可對應虛擬獎勵顯示，不一定要做完整商城，但至少要有「可累積、可兌換、可展示」的概念。

#### 最低可行版本（MVP）
至少包含：
1. 點數累積機制
2. 分數顯示區
3. 獎勵清單或兌換區塊
4. 可消耗點數兌換虛擬獎勵
5. 兌換後有明確回饋

#### 獎勵範例
- 勳章
- 貼紙
- 成就徽章
- 小獎盃圖示
- 虛擬收藏品

#### 實作要求
- 先做前端本地端版本即可，可使用 localStorage 保存點數與兌換狀態。
- 點數來源需清楚，避免與原始評估分數混淆。
- UI 命名要明確，例如：
  - 遊戲得分
  - 獎勵點數
  - 累積點數
- 若目前已有總分欄位，需重新命名避免誤解。

#### 驗收標準
- 至少一個遊戲可獲得點數。
- 點數可被累積與顯示。
- 使用者可以執行至少一種兌換動作。
- 點數系統與原本評分邏輯可區分。

## 建議實作順序
1. 先盤點專案結構與遊戲共用元件
2. 調整 SPMSQ 題庫資料結構
3. 重構遊戲頁面版型與共用樣式
4. 修改打地鼠節奏與提示動畫
5. 調整時間型遊戲設定與顯示格式
6. 建立點數／獎勵系統
7. 串接各遊戲得分來源
8. 進行整體 UI 測試與手動驗收

---

## 程式實作原則

### 維護性
- 避免把新邏輯散落在多處重複程式碼
- 優先抽成共用設定、共用工具函式或共用樣式
- 變數命名清楚，避免魔術數字

### 相容性
- 不要引入過重框架
- 以目前專案技術棧為主
- 若專案原本為純前端 HTML/CSS/JS，就維持原風格整合

### 長者友善
- 字體大
- 按鈕大
- 對比清楚
- 反應節奏放慢
- 提示明確
- 避免畫面太複雜

## 希望 Codex 交付的內容
請直接修改專案並提供：

1. **變更摘要**
   - 說明修改了哪些檔案
   - 每項修改對應哪一條需求

2. **實作內容**
   - 完成上述 1～5 項需求
   - 若某項需求受限於現有架構，請提出最接近需求的可行實作

3. **可調參數清單**
   - 例如打地鼠提示秒數、地鼠停留秒數、遊戲時間長度、點數規則等
   - 說明這些參數在哪個檔案、哪個物件

4. **驗收說明**
   - 如何操作才看得到每項修改效果
   - 哪些頁面或遊戲需特別測試

5. **若有新增資料結構**
   - 說明格式與用途
   - 特別是 SPMSQ 題庫、獎勵設定、點數儲存格式

## 額外要求
- 不要只做表面文字修改，需真正調整互動節奏與 UI。
- 不要破壞既有可用功能。
- 優先做出可直接 demo 的版本。
- 若有必要，幫忙補上簡單註解，讓後續維護容易。
- 若某些遊戲共用同一套時間／得分機制，請盡量整合，而不是各自寫一份。

---

## 完成定義（Definition of Done）
以下條件都滿足才算完成：

- SPMSQ 題目已改為一般常識導向
- 遊戲畫面更大，且標題／返回按鈕整合到遊戲畫面內
- 打地鼠速度變慢，且出洞前有清楚提示
- 時間型遊戲延長為分鐘級
- 新增獨立點數與獎勵兌換機制
- 畫面可正常操作，沒有明顯壞版
- 有提供變更摘要與驗收說明
